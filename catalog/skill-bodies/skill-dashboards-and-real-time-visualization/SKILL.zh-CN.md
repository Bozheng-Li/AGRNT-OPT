# 仪表板与实时可视化

## 概述

当可视化是一个系统而不是一张截图时，请使用此技能。这意味着更新节奏、延迟、交互设计、可观测性和渲染预算与图表选择同等重要。

如果主要请求涉及如何测试实时仪表板、冻结数据流、模拟刷新行为，或端到端覆盖告警和陈旧状态，请先路由到 `../testing-data-visualizations/SKILL.md`。

除非明确排除，否则默认考虑移动端运营使用。使用 `../../references/foundations/mobile-first-responsive-visualization.md` 规划移动端竖屏仪表板、可选横屏模式、触控交互、屏幕键盘行为、网络不稳定处理，以及告警或振动策略。

此技能涵盖四个紧密相关的问题：

- 实时流式传输和刷新行为
- 布局层级与扫描优先的仪表板构图
- 交互模式与协调视图
- 性能与规模

## 默认问题

1. 更新模型是什么？
   - 仅追加数据流
   - 定期轮询
   - 事件突发
   - 完整快照替换
2. 哪种延迟至关重要？
   - 亚秒级监控
   - 近实时运营审查
   - 异步报告
3. 用户必须做什么？
   - 注意异常
   - 比较当前与历史状态
   - 检查原因
   - 筛选和下钻
4. 硬性限制是什么？
   - 帧预算
   - 内存预算
   - 网络预算
   - 可导出性
   - 移动端带宽、电池和热预算
5. 可以同时显示多少个可视化实例？
   - 单个焦点图表
   - 少量协调面板
   - 许多重复图块或迷你折线图
6. 哪种移动端状态必须能在中断情况下工作？
   - 网络不稳定或离线
   - 应用进入后台或标签页可见性改变
   - 单手使用
   - 告警确认
   - 用于筛选或备注的屏幕键盘

## 实时设计规则

- 清晰展示时间窗口。
- 区分实时数据与历史上下文。
- 明确处理缺失、迟到和乱序事件。
- 在重连期间保持最后一个已知正常可视化可见，并将陈旧、延迟、部分、离线和错误状态与正常实时数据明确区分。
- 在证据附近显示最后更新时间和更新节奏。
- 在倾倒原始点之前使用聚合、降采样或汇总。
- 提供告警阈值、注释和状态转变，而不只是移动的线条。
- 仅针对用户请求且有意义的状态变化使用振动、系统通知或推送式告警；始终提供应用内视觉和无障碍告警路径。
- 保持最重要指标的位置和编码稳定。
- 让图例与其图表相邻：直接标签或图表旁图例优于放在屏幕其他位置的共享图例。
- 当紧凑的趋势上下文有助于扫描时，使用迷你折线图和其他微型图表，但依靠邻近标签和值来承载含义。
- 围绕态势感知设计仪表板，而不是追求最多图块数量。
- 优先使用不言自明的面板和简洁标签，而不是散落在 UI 各处的说明段落。

## 布局与扫描默认值

- 为屏幕提供清晰的焦点路径：先显示当前状态，然后是辅助上下文，最后是控件和次要诊断。
- 除非用户确实需要在同类对象间均匀扫描，否则避免每个图块视觉权重相同的网格。
- 将筛选器和开关放在它们所影响的视图附近，而不是把所有控件集中到远处的控件栏。
- 仅为异常、限定条件或操作保留标注和叙事文本。正常状态无需教程文字也应清晰可读。
- 折叠或延后低价值控件，让实时状态在视觉上保持主导。
- 在移动端，不要把筛选器栏或诊断控件堆叠在实时状态之前。使用底部抽屉、侧边抽屉、标签页或行内控件，并在“应用”“取消”“重置”或关闭后让用户回到受影响的可视化。
- 当宽时间线、地图、场地、路线、密集表格或多序列轨迹在手持设备宽屏方向下明显更清晰时，对监控视图使用移动端横屏，但仍需提供竖屏摘要。

## 交互模式

- 先看概览，再筛选或聚焦，最后按需查看详情
- 概览加详情
- 焦点加上下文
- 刷选与联动
- 悬停用于预览，点击用于确认
- 在触控设备上，轻触/聚焦用于预览，再次轻触或显式操作用于确认
- 下钻与穿透
- 更新后仍保留的持久选择
- 显式重置和撤销路径

交互应降低认知负担，而不是把关键上下文隐藏在持续的鼠标移动之后。

对于移动端仪表板，为密集标记添加逐项浏览控件、搜索或最近项选择；不要依赖悬停、像素级精确轻触，或会困住页面滚动的单指图表平移。

## 性能默认值

1. 仅在确实需要时才为 16 ms 帧时间制定预算。
2. 优化代码之前先减少工作量：
   - 更少的标记
   - 更智能的聚合
   - 更小的重绘区域
   - 在适当情况下采用更低频率的更新
3. 有意选择渲染器：
   - SVG 用于语义和注释
   - Canvas2D 用于密集 2D 栅格工作负载
   - WebGL、deck.gl、PixiJS、Sigma.js 或 Three.js 用于 GPU 规模标记、地图、粒子、图渲染或真正的 3D
4. 对长期运行的实时视图使用环形缓冲区、视口剔除和多分辨率摘要。
5. 将交互状态与渲染状态分离，使更新保持可预测。
6. 仅将粒子或辉光用于仪表板中的活动流动、告警状态、焦点或新近程度。避免与监控争夺注意力的环境动态效果。

## 输出要求

- 描述更新模型和故障模式。
- 描述移动端重连、陈旧数据、离线/部分数据和低带宽行为。
- 说明用户在触碰任何控件之前，首次扫描时应理解什么。
- 说明移动端用户首先看到什么，以及主可视化如何在控件和键盘输入周围保持可见。
- 构建控件之前定义交互契约。
- 说明当数据速率或标记数量上升时，系统如何降级。
- 对于新工作，加入技术设计章节，涵盖同时存在的图表数量、单实例与页面级预算，以及所选渲染策略的维护影响。
- 明确呈现无障碍和导出策略。
