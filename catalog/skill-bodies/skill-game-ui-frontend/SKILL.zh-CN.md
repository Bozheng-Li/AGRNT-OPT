# 游戏 UI 前端

## 概述

只要游戏需要可见的界面层，就应使用此技能。任务不是制作通用的仪表盘 UI，而是制作可读、符合主题，并能为游玩体验提供支持的浏览器游戏界面。

默认假设：使用 canvas 或 WebGL 构建游戏世界，使用 DOM 构建文本密集型 UI。

## 前端标准

1. 编码前先确定视觉方向。
   - 类型与幻想设定
   - 材质语言
   - 字体排印
   - 调色板
   - 动效基调
2. 使用 CSS 变量定义 UI 主题。
3. 建立清晰的层级。
   - 首先呈现关键的战斗或生存信息
   - 其次呈现辅助工具
   - 将不常用的设置放在菜单或抽屉后面
4. 首先保护游戏场地，尤其是在 3D 中。
   - 初始屏幕应让人感觉几秒内即可开始游玩。
   - 默认只保留一个主要的常驻 HUD 信息组，最多再加一个小型次要信息组。
   - 正常游玩时，保持游戏场地中央清爽。
   - 正常游玩时，尽量保持游戏场地中下部清爽。
   - 将背景故事、现场笔记、任务详情和冗长的控制列表放在抽屉、开关或暂停界面后面。
   - 优先使用情境提示和短暂提示，而非永久显示的方框面板。
5. 确保叠加层在运动画面之上仍清晰可读。
   - 必要时使用衬底面板、边缘处理、对比度和克制的模糊效果。
6. 从一开始就同时面向桌面端和移动端设计。
7. 围绕摄像机与输入控制的边界设计 3D UI。
   - 菜单、对话框或指针驱动的 UI 处于活动状态时，暂停或拦截摄像机控制输入。
   - 明确定义指针锁定、拖动查看和菜单交互状态。

## 3D 起步默认方案

对于探索、移动或第三人称起步脚手架，优先采用以下 UI 预算：

- 边缘处放置一个紧凑的目标标签或状态条
- 一个短暂显示的控制提示或交互提示
- 一个可选、可折叠的次要界面，例如日志、地图或任务记录

不要在首次加载时打开所有信息界面。用户应在打开任何更深层 UI 之前就能读懂场景。

作为 3D 浏览器游戏的默认实现约束：

- 不要采用“常驻全宽页眉 + 多卡片主体 + 全宽页脚”的布局
- 正常移动时，不要在屏幕中央或中下部放置大型叠加层
- 除非用户明确要求更密集的布局，否则桌面端常驻 HUD 覆盖的视口面积不要超过约 20–25%
- 在移动端，应先折叠为窄栏堆叠或情境标签，再考虑用较大面板覆盖游戏场地

## 提示词规则

要求模型设计或实现游戏 UI 时，应包括：

- 游戏幻想设定
- 摄像机或视角
- 玩家的动作
- HUD 层级
- 游戏为 3D 时的摄像机或控制模式
- 动效基调
- 桌面端和移动端预期
- 游戏场地保护与渐进披露策略
- 明确要避免的反模式

具体的提示词结构见 `../../references/frontend-prompts.md`。

## 动效规则

- 相比持续不断的微动画，应优先采用少量但有意义的转场。
- 强烈动效应保留给状态变化、奖励、危险和新手引导。
- 对非必要动画，应尊重减少动态效果的设置。
- 避免 3D HUD 动效与摄像机运动争夺注意力。

## 优秀成果的特征

- HUD 元素清晰易读，同时不会削弱场景的空间感。
- 菜单感觉属于游戏世界，而不像 SaaS 管理面板。
- 布局能在不同断点间流畅适配。
- 指针、键盘和游戏状态反馈都清晰明确。
- 在 3D 游戏中，菜单和 HUD 状态不会与摄像机控制或指针锁定冲突。
- 在 3D 游戏中，首个可玩视图将大部分视口留给移动、瞄准和空间理解。
- 常驻信息的密度足够低，使截图看起来仍像游戏场景，而非 UI 设计稿。

## 反模式

- 通用应用仪表盘布局
- 没有主题的扁平占位样式
- 未经设计考量的默认字体栈
- 遮挡游戏场地的密集叠加层
- 覆盖在实时可玩场景上的大型标题卡或多段说明
- 沿视口每一条边分布、视觉权重相同的方框面板
- 首次加载时同时展开控制、目标、笔记和背景故事
- 在 3D 游玩中使用全宽的上下框架，并在中央或主体区域常驻大型面板
- 每个元素都有过量动效
- DOM 本可更清晰、更低成本时，仍使用纯 Canvas UI
- 标准 DOM 本可更清晰时，强行将 HUD 控件置入 3D 场景
- 模态框或物品栏面板打开后仍让摄像机输入保持活动
