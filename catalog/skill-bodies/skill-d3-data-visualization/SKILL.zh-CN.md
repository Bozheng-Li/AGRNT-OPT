# D3 数据可视化

## 概述

对于重视 SVG 或 DOM 语义、且声明式语法已无法自然适配的自定义浏览器可视化，请使用此技能。当图表需要定制比例尺、标记、布局、过渡、缩放、刷选、注释或矢量导出时，D3 最具优势。

默认假设：使用 D3 处理比例尺、布局、几何、标签、注释层和行为。如果已有框架负责周围的 UI，不要把 D3 变成整个应用程序架构。

## 适合选择 D3 的情况

- 图表需要自定义标记或非标准布局
- SVG 质量对导出、打印或无障碍很重要
- 注释和标注是设计的一部分，而不是事后补充
- 可视化需要自定义矢量背景几何，例如场地、球场、赛道、平面图、示意图或其他有意义的上下文界面
- 编辑式叙事需要数据绑定的生成式剪裁图、插画基底、滚动叙事/视差状态，或 SVG 中的动画注释呈现
- 标记数量适中，DOM 或 SVG 足以承载
- 交互模型需要缩放、刷选、拖拽或协调视图
- 在 `../uml-and-software-architecture-visualization/SKILL.md` 已定义图示语义后，类 UML、依赖关系、架构、状态或流程图还需要定制 SVG 注释或产品专用构图
- 声明式语法会比生成的 D3 代码更难阅读

## 不适合优先使用 D3 DOM 渲染的情况

- 标记数量太多，以至于 DOM 吞吐量成为瓶颈
- 动画持续运行且帧预算紧张
- 使用 Canvas2D 或 WebGL 能更快解决任务

## 工作模式

1. 首先构建干净的数据模型。
2. 分离以下部分：
   - 解析与归一化
   - 比例尺构建
   - 派生几何
   - 渲染
   - 交互状态
3. 在数据连接中优先使用稳定键。
4. 使用 D3 处理数学和行为，而不是掩盖薄弱的状态管理。
5. 评估页面上可能共存多少个 D3 实例，以便在仪表板规模下评估 DOM、布局和事件成本。
6. 如果使用上下文界面，请明确源单位和渲染比例尺，将背景绘制为独立图层，并使标记放置或布局力适应领域几何。
7. 对于经过艺术指导的叙事，请将生成式图像、基底、数据标记、标签和注释层彼此分离，以便分别审查和导出。
8. 保持注释和交互覆盖层显式可见。
9. 对于 SVG 输出，在宣布图表完成前应用 `./references/svg-polish-and-crispness.md`。明确设置字体大小、刻度间距、网格线描边、数据描边宽度、图标尺寸、标签对齐和缩放时稳定的描边行为。
10. 对于面向浏览器的工作，使用 `../../references/foundations/mobile-first-responsive-visualization.md`，让 D3 布局针对大屏和移动端状态重新计算，而不是从单一桌面端 SVG 缩小。

## 编辑式默认值

- 构建显式标签层。优先考虑直接末端标签、行内图例或面板标签，再考虑分离式图例。
- 构建显式注释层。标注应附着于数据点、范围、阈值或领域几何，而不是作为装饰漂浮。
- 让网格线、坐标轴和参考标记保持低调。大部分墨水应属于数据、标签或注释。
- 默认 SVG 图表精修参数：坐标轴刻度文字 10–12 px，直接标签 11–13.5 px，非数据描边 0.5–1 px，普通数据线 1.5–2.25 px，焦点线 2.5–3 px，短刻度线 4–6 px；除非粗重图表边框承载含义，否则不要使用。
- 将 D3 坐标轴用作几何生成器，然后设置样式。优先使用 `.tickSizeOuter(0)`、6–8 px 的刻度间距、低调网格线，在多余时移除轴域，并在旋转标签或使用极小标签前先减少刻度数。
- 对笔直坐标轴、网格线和矩形单元格使用 `shape-rendering: crispEdges`；不要全局应用于曲线、符号、圆形或对角标记。
- 对可缩放轮廓、坐标轴、注释连接线、地图边界和应保持屏幕描边宽度稳定的图标使用 `vector-effect: non-scaling-stroke`。
- 仅当叙事需要时才使用自定义布局；不要复制出版物的构图或视觉标识。
- 对于窄屏宽度，应提供简化的标记布局、堆叠式分面小图，或图表下方的编号注释图例。
- 在移动端，将轻触/焦点作为检查路径，将命中区域扩大到可见标记之外，提供拖拽替代方案，定义捏合或缩放的所有权，并在筛选器或设置打开时保持主图表可见。
- 对需要导出、无障碍和精确标签的编辑式图表，优先使用可检查的 SVG。仅当 DOM 数量足以证明有必要时，才将密集标记场迁移到 Canvas。
- 对于滚动叙事或视差，使用 `../scrollytelling-and-parallax-data-visualization/SKILL.md` 处理滚动叙事，然后让 D3 负责比例尺、派生几何、数据连接、标签和 SVG 过渡。确保每个场景作为静态画面也成立。
- 对于生成式图像，使用 D3/SVG 处理标签、锚点、遮罩、裁剪和数据覆盖层，而不是把数字烘焙进图像。
- 对于动态效果，使用过渡展现顺序、方向、累积、比较或机制。通过渲染最终状态来尊重减少动态偏好。
- 对于看起来像 3D 的 SVG，除非透视失真表达真实的多轴表面，否则应避免；标注坐标轴，并在需要时提供平面后备方案。

## D3 核心构建模块

- `d3-array` 用于分组、范围、刻度和汇总
- `d3-scale` 用于位置、颜色和大小映射
- `d3-axis` 用于可读坐标轴
- `d3-shape` 用于生成线、面积、弧和符号
- `d3-selection` 和 `d3-transition` 用于 DOM 行为
- `d3-brush`、`d3-zoom` 和 `d3-drag` 用于直接操控
- 仅当数据结构适合时使用 `d3-force`、`d3-hierarchy`、`d3-contour` 和其他布局

## 集成规则

- 在 React 或其他 UI 框架中，优先让框架负责结构，让 D3 负责几何、比例尺和行为。
- 对于高度定制的图表，只要边界清晰，小型命令式 D3 岛也是合适的。
- 将响应式行为绑定到容器测量值，而不是魔法常量。
- 针对移动端竖屏和可选横屏尺寸重新计算标签、刻度数量、注释放置、碰撞规则和绘图区高度；不要只调整外层 `viewBox` 的大小。
- 即使在混合系统中，也优先使用 SVG 处理标签和注释。
- 除非上下文背景层参与选择、缩放或命中测试，否则保持其不可交互。
- 当 DOM 数量或持续重绘成为限制因素时，迁移到 Canvas 或 WebGL。

## 输出要求

- 解释为什么 D3 是正确的渲染层。
- 保持图表结构具有语义且可检查。
- 需要时提供 SVG 或 PNG 导出路径。
- 为交互配备明显的注释和重置控件。
- 对于移动端，说明触控目标策略、悬停替代方式、拖拽/捏合所有权，以及屏幕键盘或设置面板如何让用户返回图表。
- 绘制上下文界面时，说明源几何，以及数据标记如何受其定位、约束或偏置。
- 对于图像辅助的编辑式工作，说明哪些素材是生成的、哪些图层仍与数据绑定，以及标签如何锚定到基底。
- 对于动画，说明动作、场景状态、时长预算和减少动态后备方案。
- 对于新工作，加入技术设计章节，涵盖同时存在的实例数量、DOM 和交互成本，以及与声明式或基于 Canvas 的替代方案相比的维护权衡。
