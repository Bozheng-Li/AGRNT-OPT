# 系统化调试

## 概述

随机尝试修复会浪费时间并制造新缺陷。快速补丁会掩盖底层问题。

**核心原则：** 尝试修复前，始终先找出根因。只修症状就是失败。

**违背这套流程的字面要求，就是违背调试的根本精神。**

## 铁律

```
NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST
```

没有完成阶段 1，就不能提出修复方案。

## 何时使用

适用于任何技术问题：
- 测试失败
- 生产环境缺陷
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**以下情况尤其应使用：**
- 面临时间压力（紧急情况最容易诱发猜测）
- “只做一个快速修复”看起来显而易见
- 已尝试过多个修复
- 上一次修复没有奏效
- 尚未完全理解问题

**以下情况也不要跳过：**
- 问题看似简单（简单缺陷同样有根因）
- 正在赶时间（仓促行事必然返工）
- 管理者要求立刻修好（系统化方法比胡乱尝试更快）

## 四个阶段

必须完成当前阶段，才能进入下一阶段。

### 阶段 1：根因调查

**尝试任何修复之前：**

1. **仔细阅读错误消息**
   - 不要跳过错误或警告
   - 它们往往包含确切的解决线索
   - 完整阅读堆栈跟踪
   - 记下行号、文件路径和错误码

2. **稳定复现**
   - 能可靠触发问题吗？
   - 精确步骤是什么？
   - 每次都会发生吗？
   - 如果无法复现 → 收集更多数据，不要猜测

3. **检查近期变更**
   - 哪些变化可能导致此问题？
   - 查看 Git diff 和近期提交
   - 检查新依赖、配置变更
   - 检查环境差异

4. **在多组件系统中收集证据**

   **当系统包含多个组件（CI → 构建 → 签名，API → 服务 → 数据库）时：**

   **提出修复前，先添加诊断观测：**
   ```
   For EACH component boundary:
     - Log what data enters component
     - Log what data exits component
     - Verify environment/config propagation
     - Check state at each layer

   Run once to gather evidence showing WHERE it breaks
   THEN analyze evidence to identify failing component
   THEN investigate that specific component
   ```

   **示例（多层系统）：**
   ```bash
   # Layer 1: Workflow
   echo "=== Secrets available in workflow: ==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # Layer 2: Build script
   echo "=== Env vars in build script: ==="
   env | grep IDENTITY || echo "IDENTITY not in environment"

   # Layer 3: Signing script
   echo "=== Keychain state: ==="
   security list-keychains
   security find-identity -v

   # Layer 4: Actual signing
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **这样会揭示：** 究竟是哪一层失败（机密 → 工作流 ✓，工作流 → 构建 ✗）。

5. **跟踪数据流**

   **当错误位于调用栈深处时：**

   完整的反向追踪方法见本目录中的 `root-cause-tracing.md`。

   **快速版本：**
   - 错误值源自哪里？
   - 谁用错误值调用了这里？
   - 持续向上追踪，直到找到源头
   - 修复源头，而不是症状

### 阶段 2：模式分析

**修复前先找出模式：**

1. **寻找正常工作的示例**
   - 在同一代码库中寻找相似且正常工作的代码
   - 与故障代码相似的正常实现做了什么？

2. **对照参考实现**
   - 如果要实现某种模式，必须完整阅读参考实现
   - 不要略读——阅读每一行
   - 完全理解模式后再应用

3. **识别差异**
   - 正常实现与故障实现有何不同？
   - 列出所有差异，无论多么细小
   - 不要假设“这不可能有影响”

4. **理解依赖**
   - 它还需要哪些组件？
   - 需要哪些设置、配置和环境？
   - 它依赖哪些假设？

### 阶段 3：假设与测试

**使用科学方法：**

1. **提出单一假设**
   - 清楚说明：“我认为 X 是根因，因为 Y”
   - 将它写下来
   - 具体明确，不要含糊

2. **最小化测试**
   - 用尽可能小的变更检验假设
   - 一次只改变一个变量
   - 不要同时修复多个问题

3. **继续前先验证**
   - 奏效了吗？是 → 阶段 4
   - 没奏效？提出一个新假设
   - 不要在原有尝试之上继续叠加修复

4. **不知道时**
   - 明确说“我不理解 X”
   - 不要假装知道
   - 寻求帮助
   - 做更多研究

### 阶段 4：实施

**修复根因，而不是症状：**

1. **创建失败测试用例**
   - 使用最简单的复现方式
   - 尽可能写自动化测试
   - 没有测试框架时写一次性测试脚本
   - 修复前必须具备该用例
   - 使用 `superpowers:test-driven-development` 技能编写恰当的失败测试

2. **实施单一修复**
   - 处理已经识别的根因
   - 一次只做一个变更
   - 不做“既然来了就顺手改”的优化
   - 不夹带重构

3. **验证修复**
   - 测试现在通过了吗？
   - 其他测试有没有被破坏？
   - 问题真的解决了吗？

4. **如果修复无效**
   - 停止
   - 统计已经尝试了多少次修复
   - 如果少于 3 次：回到阶段 1，利用新信息重新分析
   - **如果达到或超过 3 次：停止并质疑架构（见下面第 5 步）**
   - 未进行架构讨论前，不要尝试第 4 个修复

5. **如果 3 次以上修复均失败：质疑架构**

   **表明存在架构问题的模式：**
   - 每次修复都会在不同位置暴露新的共享状态、耦合或问题
   - 每个修复都需要“大规模重构”才能实施
   - 每次修复都会在其他位置制造新症状

   **停止并质疑基本前提：**
   - 这种模式从根本上合理吗？
   - 我们是否“只是凭惯性继续坚持”？
   - 应当重构架构，还是继续修补症状？

   **尝试更多修复前，先与你的人类协作者讨论。**

   这不是假设失败，而是架构本身错误。

## 危险信号——停止并遵循流程

如果发现自己正在这样想：
- “先快速修好，以后再调查”
- “先改一下 X，看看是否有效”
- “同时改几处，再运行测试”
- “跳过测试，我手动验证”
- “大概是 X，我来修掉它”
- “虽然不完全理解，但这样可能有效”
- “模式要求 X，但我会用另一种方式调整”
- “主要问题如下：[未调查就列出修复项]”
- 尚未跟踪数据流就提出解决方案
- **“再试一次修复”（已经尝试过 2 次以上）**
- **每次修复都会在另一位置暴露新问题**

**以上每一种情况都意味着：停止。回到阶段 1。**

**如果 3 次以上修复失败：** 质疑架构（见阶段 4.5）。

## 你的人类协作者发出的错误信号

**留意以下纠正：**
- “不是这样吗？”——你没有验证就作了假设
- “这能让我们看到……吗？”——你本应先添加证据收集
- “别再猜了”——你尚未理解问题就提出修复
- “深入想想”——质疑基本前提，而不只是症状
- “我们卡住了吗？”（感到挫败）——当前方法没有奏效

**看到这些信号时：** 停止。回到阶段 1。

## 常见的自我辩解

| 借口 | 事实 |
|--------|---------|
| “问题很简单，不需要流程” | 简单问题也有根因。对简单缺陷而言，这套流程同样很快。 |
| “情况紧急，没时间走流程” | 系统化调试比反复猜测和尝试更快。 |
| “先试试这个，再去调查” | 第一次修复会定下模式。从一开始就用正确方式。 |
| “确认修复有效后再补测试” | 未经测试的修复无法持久。测试先行才能证明效果。 |
| “一次改多个地方更省时间” | 无法分辨哪个改动起效，还会制造新缺陷。 |
| “参考资料太长，我自行调整模式” | 一知半解必然产生缺陷。完整阅读它。 |
| “我看出问题了，让我来修” | 看到症状 ≠ 理解根因。 |
| “再试一次修复”（已失败 2 次以上） | 3 次以上失败 = 架构问题。质疑模式，不要再修一次。 |

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|-------|---------------|------------------|
| **1. 根因** | 阅读错误、复现、检查变更、收集证据 | 理解发生了什么以及为什么发生 |
| **2. 模式** | 寻找正常示例并比较 | 识别差异 |
| **3. 假设** | 提出理论并进行最小化测试 | 假设得到确认，或形成新假设 |
| **4. 实施** | 创建测试、修复、验证 | 缺陷已解决，测试通过 |

## 当流程得出“没有根因”时

如果系统化调查表明问题确实由环境、时序或外部因素引起：

1. 说明你已经完成这套流程
2. 记录调查过的内容
3. 实施合适的处理方式（重试、超时、错误消息）
4. 添加监控或日志，供将来调查

**但是：** 95% 的“没有根因”案例只是调查不完整。

## 支持技术

以下技术属于系统化调试的一部分，并可在本目录中找到：

- **`root-cause-tracing.md`**——沿调用栈向后追踪缺陷，找出原始触发点
- **`defense-in-depth.md`**——找到根因后，在多个层次添加验证
- **`condition-based-waiting.md`**——以条件轮询替换任意超时

**相关技能：**
- **superpowers:test-driven-development**——用于创建失败测试用例（阶段 4，第 1 步）
- **superpowers:verification-before-completion**——在宣称成功前验证修复确实有效

## 实际影响

来自调试会话的数据：
- 系统化方法：15–30 分钟完成修复
- 随机修复：反复折腾 2–3 小时
- 首次修复成功率：95% 对 40%
- 引入新缺陷：几乎为零，对比常见发生
