# Gantt 图可视化

## 概述

当工作围绕时间跨度、里程碑、依赖关系、资源或进度风险组织时，使用此技能。Gantt 图既是一种图表，也是一种产品界面：它通常结合任务表格、日历轴、任务条、里程碑、依赖连线、编辑规则，以及与项目管理事实来源的集成。

默认假设：只有当进度计划本身就是证据时，才推荐 Gantt 图。如果用户主要需要了解流程状态、任务负责人、排序、问题状态或日程预订，应先考虑 Kanban、表格、里程碑时间线、依赖图、资源时间线或不确定性视图。

Gantt 图天生较宽。使用 `../../references/foundations/mobile-first-responsive-visualization.md`，使移动端竖屏提供可用的摘要或聚焦切片；当横向查看计划很重要时，还应考虑移动端横屏。

## 核心工作流

1. 对进度计划问题进行分类：
   - 计划进度、实际生命周期时间线、路线图、资源计划、产能视图、基线偏差、关键路径审查或利益相关方快照
   - 用户需要的是只读说明、探索性分析，还是可编辑的项目规划
2. 设计前检查数据源：
   - 真正的排程引擎、任务跟踪器、路线图视图、资源日历、静态导出，还是可视化制品
   - 原生字段、自定义字段、仅日期值与日期时间值、时区策略、层级、依赖语义、日历和权限
   - 每个映射字段及任何推断值的来源证据
3. 规范化为 Gantt 模型：
   - 任务、层级或 WBS、开始、结束、工期、进度、状态、负责人或资源、里程碑、依赖关系、基线、日历、约束、估算、实际值、源 ID 和来源置信度
4. 判断 Gantt 是否为正确界面：
   - 当时间跨度以及依赖或资源推理驱动决策时，使用 Gantt
   - 对只有少量日期的高管摘要，使用里程碑时间线
   - 对工作流状态和吞吐量，使用 Kanban
   - 当查找和精确字段占主导时，使用表格
   - 当结构比日期更重要时，使用依赖图
   - 对没有项目依赖关系的预订，使用日历或资源时间线
   - 当日期风险具有概率性或估算仍不稳定时，使用不确定性视图
5. 设计默认视图：
   - 冻结任务表格并搭配日历轴
   - 今日标记、可见时间尺度，以及在有意义时显示周末或非工作时间
   - 清晰的任务条、里程碑、依赖连线、基线、进度，以及关键路径或风险样式
   - 无需悬停即可理解的行分组、层级和直接标签
   - 移动端竖屏摘要或聚焦默认视图，不能把每一行都缩小到难以辨认
   - 当读取宽时间线、追踪依赖关系或编辑很重要时，定义移动端横屏行为
6. 定义交互契约：
   - 缩放和平移、行虚拟化、展开/折叠、搜索、筛选、排序、悬停预览、确认选择、键盘导航、拖动/调整大小、依赖关系编辑、撤销/重做、导出和深层链接
   - 移动端的触控目标、拖动替代方式、键盘打开时的搜索/筛选行为，以及从设置返回后的行为
7. 根据真实规模选择渲染器和库：
   - 行数、可见时间范围、依赖关系数量、可编辑性、实例数量、导出要求，以及与源系统同步的需求
8. 规划测试和发布门禁：
   - 数据适配器测试、日期/时间夹具、依赖图验证、视觉状态、无障碍、导出和端到端排程工作流

## 何时使用 Gantt

- 具有开始和完成日期的任务型项目计划。
- 工作分解结构、阶段、发布、Epic、施工计划、制造计划、实施计划、上线计划和迁移切换。
- 延误会影响下游工作的依赖链。
- 关键路径、浮动时间、基线、截止日期或偏差分析。
- 随时间变化的资源分配、产能规划或工作负载冲突。
- 当问题是“什么在何时发生”，而非“它处于什么状态”时的跨团队路线图。
- 从 MS Project、Primavera P6、Jira Advanced Roadmaps、GitHub Projects、Smartsheet、monday.com、Asana、ClickUp、Azure DevOps 或 CSV/XLSX 导出文件导入进度计划。

## 何时不应使用 Gantt

- 规模较小或高度流动的工作，此时简单列表、表格或 Kanban 看板更清晰。
- 状态、优先级、负责人或 SLA 比计划跨度更重要的问题队列。
- 只有发布或上线日期的高管摘要；应使用里程碑时间线或路线图。
- 没有任务依赖关系的预订、预约或排班；应使用日历或资源时间线。
- 没有可靠日期的纯依赖关系推理；应使用图或矩阵。
- 以不确定性为核心叙事的概率进度或早期估算；应使用区间、情景或风险视图。
- 仅基于创建、开始和关闭时间戳的实际生命周期分析；除非用户明确要求查看实际流转历史而非计划进度。

## 技术栈选择

- 企业级可编辑计划：当依赖关系编辑、日历、基线、关键路径、资源视图、撤销、导入/导出和排程规则属于产品要求时，使用 Bryntum Gantt、DHTMLX Gantt、Kendo UI Gantt、Syncfusion Gantt 或同类排程组件。
- 轻量展示或报告：当视图以只读为主且进度计划语义已计算完成时，使用 Highcharts Gantt、Frappe Gantt、Plotly timelines、Observable Plot 或类似工具。
- 资源预订或团队可用性：当资源和日历时段比 WBS 排程更重要时，使用 FullCalendar resource timeline 或 scheduler 类组件。
- 小型编辑型计划：当标签、注释、矢量导出和精确编排很重要时，使用 D3/SVG 或声明式语法。
- 大型密集产品界面：使用虚拟化的混合布局，以 HTML 呈现表格文本，以 SVG 或 Canvas 呈现任务条和依赖连线；当 DOM 成本占主导时，用 Canvas 呈现密集的任务条/连线层。
- 除非标记数量、连续平移/缩放、GPU 拾取或自定义着色器效果使 Canvas 和 DOM 无法胜任，否则避免直接使用 WebGL。

## 输出要求

- 说明 Gantt 是否为首选建议；若不是，则说出更合适的替代方案。
- 说明数据源是计划进度数据、实际生命周期数据、路线图快照、资源日历，还是静态可视化制品。
- 指明哪些字段可信、经过映射、由推断得出或缺失。
- 包含规范的进度计划模型、最低交互要求、技术栈选择、性能假设、无障碍计划和导出路径。
- 对外部数据源，保留源 ID，并明确指出含义模糊的自定义字段，不要悄然将其视为计划事实。
- 对可编辑计划，说明拖动、调整大小、依赖关系变更、日历、基线和同步失败时的验证与冲突处理行为。
- 对移动端，说明竖屏摘要或聚焦切片、必要时的横屏支持、触控/编辑替代方式、键盘行为，以及远程计划的离线/过期同步行为。
