# 地理空间与地图制图可视化

## 概述

当地点、投影、移动或空间邻接关系对分析很重要，或者产品界面确实需要地图交互模型时，使用此技能。不要仅仅因为数据集中有经度、纬度或地区名称，就条件反射地使用地图。

默认假设：只有当地理本身参与推理，或产品确实需要地图界面时，才使用地图。如果任务是比较、排序或展示地区间的分布，非地图视图可能更清晰。尽早区分分析制图与产品地图 UX。

移动端地图用户十分常见，而且往往是主要用户。在选择标记密度、标签策略、竖屏/横屏行为、触控手势、地理定位/摄像头/AR 能力的使用方式，以及处理不稳定的图块或数据连接时，使用 `../../references/foundations/mobile-first-responsive-visualization.md`。

## 工作模式

1. 判断问题具有空间属性，还是仅按地点分组。
2. 提问前先推断用户的地图用途。推荐一个可能适用的基础承载形式并说明假设，然后只询问那些缺失且会改变地图设计的高影响问题。
3. 判断需要哪种地图体验：
   - 专题分析地图
   - 用于探索或分层上下文的可平移缩放地图
   - 路线、地点或产品地图体验
   - 地形、影像或运营态势感知地图
4. 有意识地选择底图或基础承载形式：
   - 对驾驶、路线规划、交通、配送、道路安全或驾驶天气，使用道路和城市地图
   - 对徒步、野火、洪水、山体滑坡、环境风险、户外规划或野外作业，使用地形图
   - 当地理对区域比较有意义时，使用中性的行政区划地图
   - 对卫星、遥感、损害、土地利用、农业、施工或视觉证据，使用影像或栅格基础层
   - 对密集点或事件，使用安静的中性底图；除非道路、地形或边界本身参与推理
   - 只有当深度、地形、遮挡或全球上下文会改变分析时，才使用地球仪或 3D 地形
5. 选择地图家族：
   - 分级设色图
   - 符号地图
   - 聚类符号地图
   - 流向图
   - 六边形或图块汇总
   - 栅格或密度场
   - 动画路线或行程图层
   - 融合到物理基础层上的地形、风险或损害纹理
6. 如果点发生重叠，应有意识地选择重叠处理策略：
   - 当密集点需要计数或摘要预览，并可在缩放时解聚时，使用聚类汇总符号
   - 当数量较少且每个点都需要立即可见时，使用位移或 spiderfying
   - 当大型比例符号、光环式符号或同心点符号必须在详细缩放级别保留真实坐标时，使用精确位置锚点
7. 有意识地选择多尺度缩放行为：
   - 当形状、区域或路径本身是检查对象时，使用地图空间几何体
   - 当符号、标签和标注作为定位器、徽章或类 UI 注释时，使其在屏幕上保持稳定
   - 边界、轮廓和经纬网应使用屏幕稳定的描边，除非厚度变化本身具有含义
   - 仅当跨尺度改变符号或标签大小确实能提高可读性或层级时，才使用按缩放级别分段的尺寸
8. 如果需要逐一检查密集的可见点，可考虑在当前视野或当前图层内，用键盘方向键或“上一个/下一个”控件进行逐项选择。
9. 对移动端地图，实现前先定义竖屏和可选的横屏体验：
   - 窄屏使用更少标记和更短标签
   - 必要时使用移动端专用的标记或标签变体
   - 对无法容纳的标签提供图例或详情面板
   - 为密集点提供点按、逐项浏览、搜索和重置路径
   - 当单指平移需要保留页面滚动时，使用双指缩放或显式缩放控件
   - 定义过期图块/数据、离线、低带宽和重新连接状态
10. 对科学、实时、灾害、地形、气候、地震、海洋或其他对来源敏感的空间工作，在视觉设计或实现前建立来源与方法账本。包含来源 URL/API、许可或署名、更新频率、单位、坐标参考、覆盖缺口、速率限制、缓存/回退策略，以及哪些图层是测量所得、推断所得、示意性或装饰性的。
11. 有意识地选择投影、底图样式和规范化策略。
12. 对地球仪或自定义投影，编码前检查底图/纹理、事件标记、标签、命中测试、摄像机聚焦和回退几何体之间的坐标对齐。记录经度原点、环绕策略，并至少对若干已知地点进行抽查。
13. 选择实现技术栈：
   - 对自定义投影、专题制图和注释丰富的分析地图，使用 D3 geo
   - 对轻量可平移缩放地图、标记、图层控件、GeoJSON 叠加层和熟悉的平移或缩放控件，使用 Leaflet
   - 对矢量图块样式、数据驱动样式、顾及标签的图层顺序、山体阴影、热力图、倾斜或旋转，以及现代可平移缩放地图体验，使用 MapLibre GL JS 或 Mapbox GL JS
   - 当托管道路底图、路线规划、Places、地理编码、交通上下文或 Google Maps Platform 能力驱动产品需求时，使用 Google Maps
   - 对 GIS 密集型 Web 地图、投影、WMS/WMTS/OGC 服务、矢量和栅格图层，以及高级图层控制，使用 OpenLayers
   - 对 MapLibre、Google Maps、Mapbox 或 ArcGIS 之上的高容量图层、聚合、行程、弧线、路径、类粒子流、点云、六边形或网格图层，以及 GPU 密集型地理空间交互，使用 deck.gl
   - 对托管 GIS 服务、FeatureLayer 工作流、渲染器、聚类、分箱、热力图、编辑和运营地图，使用 ArcGIS Maps SDK
   - 对企业路线规划、交通、车队、物流、地理编码、供应商数据和厂商特定位置服务，使用 Azure Maps 或 HERE Maps
   - 对 3D 地球仪、地形、影像分层、时态场景和 3D Tiles，使用 CesiumJS
14. 明确处理图例、标签、不确定性和比较需求。
15. 对编辑型地图故事，判断地图是承载证据的安静舞台、运动流场、粒子引导的路线故事、风险表面，还是滚动叙事序列。
16. 对冲突、占领、流离失所、灾害或人道主义地图，使用 `../../references/foundations/sensitive-geopolitical-and-humanitarian-stories.md`。要求提供有日期的状态、来源与方法说明、署名，并在视觉上区分测量、估算和示意几何体。
17. 如果地图接管滚轮缩放或手势平移，也必须接管浏览器行为：防止页面滚动或滚动串联，保持重置和按钮控件可见，并用鼠标滚轮、触控板、触屏平移和双指缩放验证交互。
18. 只有当 AR、摄像头、地理定位或动作传感器能实质改善空间推理或数据收集时才使用它们，并始终提供手动/无需权限的回退方案。

## 输出要求

- 说明为什么适合或不适合使用地图。
- 在选择实现技术栈前，说明推荐的底图或基础承载形式、必需的上下文图层、专题数据图层，以及任何未解决的问题。
- 指出投影、面积和规范化方面的权衡。
- 对实时/科学空间工作，包含来源与方法账本以及缺失数据/回退策略。
- 对地球仪/自定义投影，包含坐标系对齐检查，以及当选择操作移动摄像机或地球仪时的最短路径聚焦行为。
- 如果涉及大型或重叠的点符号，说明地图是否需要聚类汇总、spiderfying 或位移、精确位置锚点，或需要跨缩放级别组合这些方法。
- 对交互式地图，说明哪些图层应在地图空间中缩放，哪些应在屏幕上保持稳定。
- 对移动端地图，说明竖屏布局、横屏是否有必要、标记/标签缩减规则、触控手势归属、地理定位/摄像头/AR 的使用方式（如有），以及图块或数据过期/离线时的行为。
- 如果用户可能连续检查许多相邻点，说明除直接点击外，地图是否还需要逐项选择控件。
- 根据交互模型、数据规模、底图需求和注释要求选择技术栈。
- 对编辑型地图故事，说明视觉基础层、流向或风险图层、注释计划、动效目的和静态回退方案。
- 对动画流向图或粒子地图，说明每个运动标记代表什么、发射率是编码数量还是仅编码方向，以及如何在减少动态效果和静态导出中表达同一论点。
- 对敏感的地缘政治或人道主义地图，说明每个地图状态的日期或更新频率、来源层级、每个图层的证据状态，以及伦理表述约束。
- 指出对图块提供商、商业地图平台或样式基础设施的任何重要依赖。
